# Discovery — питання до клієнта/партнера

**Контекст:** партнер надіслав специфікацію (`Learning Platform.xlsx`) на **B2B мульти-тенантну корпоративну LMS** (50 user flows, 30 модулів, 7 ролей). Це зміна категорії продукту відносно поточного B2C single-tenant магазину курсів. Перш ніж закладати схему даних і архітектуру (особливо мульти-тенантність), треба узгодити рішення нижче.

**Як користуватись:** дописуй відповіді клієнта в рядок `**Відповідь:**` під кожним питанням. Позначай статус: 🔴 чекаємо / 🟡 частково / 🟢 узгоджено.

> ⚠️ Поточний `CLAUDE.md` (MVP «Excluded») виключає instructor / certificates / analytics / subscriptions / AI — партнер робить їх must-have. Після discovery скоуп у `CLAUDE.md` треба переписати.

## P0 — блокують архітектуру (відповісти ПЕРШИМ)

### 1. Власність контенту 🔴

*(Flow 14 «assign journey to company» → спільний; flow 49 «request custom program» → кастомний. Це визначає всю tenancy-модель контенту.)*

### 2. Користувач у кількох компаніях 🔴

*(Впливає на модель membership та ізоляцію даних.)*

### 3. Білінг і seats 🔴
- Модель: per-seat підписка, фіксована плата за компанію, чи інше?
- Самообслуговування (Stripe автоматично) чи ручний білінг (Platform Admin виставляє план)? *(Flow 38 «manual billing» схиляє до ручного — підтвердити.)*
- Seat рахується за запрошеними чи активними?
- Валюта, ПДВ/інвойси, юрособа-платник?

### 4. Ролі та права (точна матриця) 🔴
- «Instructor» — платформний чи всередині компанії? Хто перевіряє завдання (flow 23)?
- «Manager» — лише звіти по своїй команді, чи може призначати навчання?
- «Super Admin / Owner» vs «Platform Admin» — у чому різниця?
- «Company Admin» — повний контроль над своєю компанією (запрошення, команди, призначення, звіти)?
