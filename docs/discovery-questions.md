# Discovery — питання до клієнта/партнера

**Контекст:** партнер надіслав специфікацію (`Learning Platform.xlsx`) на **B2B мульти-тенантну корпоративну LMS** (50 user flows, 30 модулів, 7 ролей). Це зміна категорії продукту відносно поточного B2C single-tenant магазину курсів. Перш ніж закладати схему даних і архітектуру (особливо мульти-тенантність), треба узгодити рішення нижче.

**Як користуватись:** дописуй відповіді клієнта в рядок `**Відповідь:**` під кожним питанням. Позначай статус: 🔴 чекаємо / 🟡 частково / 🟢 узгоджено.

> ⚠️ Поточний `CLAUDE.md` (MVP «Excluded») виключає instructor / certificates / analytics / subscriptions / AI — партнер робить їх must-have. Після discovery скоуп у `CLAUDE.md` треба переписати.

## P0 — блокують архітектуру (відповісти ПЕРШИМ)

### 1. Власність контенту 🟢
Курси/програми — **спільний платформний каталог** для всіх компаній, чи кожна компанія може мати **власний/кастомний** контент? Хто авторить: лише Platform Admin/Instructor, чи Company Admin теж?
*(Flow 14 «assign journey to company» → спільний; flow 49 «request custom program» → кастомний. Це визначає всю tenancy-модель контенту.)*
**Відповідь (2026-06-15):** Авторить **Platform Admin/Instructor**; контент — **спільний платформний каталог, власність наша**. На цьому етапі компанії **не** мають кастомного контенту. Company Admin **не авторить**, але може **докуповувати додаткові курси**.
→ *Наслідок:* потрібна майбутня модель **company↔course entitlements** (які курси компанія «купила»/має право призначати). Зараз `course_assignments` дозволяє призначити будь-який `is_published` курс — це працює для пілота, але треба обмежити, коли з'явиться білінг/закупівля.

### 2. Користувач у кількох компаніях 🟢
Один email може належати кільком компаніям одночасно (консультант/підрядник), чи строго одна людина = одна компанія?
*(Впливає на модель membership та ізоляцію даних.)*
**Відповідь (2026-06-15):** Один email **може** належати кільком компаніям одночасно (бажана можливість).
→ *Наслідок:* поточна схема це **вже підтримує** — `organization_members UNIQUE(org_id, user_id)` (той самий `user_id` у різних орг ОК), дашборд агрегує призначення по всіх орг. Лишається перевірити, що ніде немає припущення «одна людина = одна компанія».

### 3. Білінг і seats 🟢
- Модель: per-seat підписка, фіксована плата за компанію, чи інше?
- Самообслуговування (Stripe автоматично) чи ручний білінг (Platform Admin виставляє план)? *(Flow 38 «manual billing» схиляє до ручного — підтвердити.)*
- Seat рахується за запрошеними чи активними?
- Валюта, ПДВ/інвойси, юрособа-платник?
**Відповідь (2026-06-15):** Модель вирішена (деталі й цифри — `docs/pricing.md`, оптимізовано з партнерської стратегії):
- **Ручний білінг** — Platform Admin активує план і виставляє `seat_limit` (вже enforced server-side). **Без self-serve Stripe для компаній** у v1.
- **Per-company план за seat-бендами**, річна оплата (місячна опційно), знижка за річну передоплату 15–20%. Per-seat — лише на запит для малих команд.
- Старт через **платний пілот** ($3.5–5k) → річний план ($6–30k/рік за seats). Валюта **CAD**.
- ⏳ Лишається: фінальні цифри/ПДВ/юрособа-платник — на підтвердження партнером (на архітектуру не впливає: модель ручна + seat-based уже реалізована).

### 4. Ролі та права (точна матриця) 🟢
- «Instructor» — платформний чи всередині компанії? Хто перевіряє завдання (flow 23)?
- «Manager» — лише звіти по своїй команді, чи може призначати навчання?
- «Super Admin / Owner» vs «Platform Admin» — у чому різниця?
- «Company Admin» — повний контроль над своєю компанією (запрошення, команди, призначення, звіти)?
**Відповідь (2026-06-15):** Спрощена матриця v1:
- **Platform Admin = Super Admin** (одна роль; `profiles.role='admin'`). Авторить контент.
- **Company Admin = Manager** (одна роль на боці компанії): запрошує працівників, докуповує курси, бачить звіти, призначає навчання.
- **Learner.**
- **Owner** — поки не потрібен. Окремих Manager / company-Instructor у v1 **немає**. Instructor — платформна авторська роль на майбутнє.
→ Збігається з поточною реалізацією (`platform_admin` / `company_admin` / `learner`).

### 5. Пріоритет і пілот 🟢
- B2B-first чи B2C одразу?
- Хто перший платний клієнт/пілот і дедлайн запуску?
- Є наявні дані/користувачі для міграції, чи з нуля?
**Відповідь (2026-06-15):** **B2B-first**, B2C пізніше. Наявних користувачів/даних для міграції **немає** — старт з нуля. Конкретний пілот-клієнт/дедлайн поки не названо.

## Найкритичніше
**P0 статус (2026-06-15):** усі 5 P0 — 🟢 узгоджено. #3 (білінг) — модель вирішена (ручний, per-company за seats, річна+знижка; `docs/pricing.md`); лишилися лише фінальні цифри/податки на підтвердження партнером (не блокує архітектуру).

**Архітектурні висновки після P0:**
- Поточна tenancy-модель (спільний каталог + org-scoped membership/assignments) — **підтверджена**, переробляти не треба.
- Мульти-компанія на один email — **уже працює** схемно; перевірити відсутність single-company припущень.
- Ролі v1 = `platform_admin` / `company_admin` / `learner` — **збігаються** з кодом.
- **Новий пункт беклогу:** company↔course **entitlements** (компанія «купує» курси) — потрібен, коли з'явиться білінг; зараз будь-який published курс призначається без обмежень.
- Білінг лишається єдиним відкритим P0 — без pricing-doc не закладаємо платіжну/seat-схему глибше за наявний seat-ліміт.
